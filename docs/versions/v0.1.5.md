# ProxyHarbor v0.1.5 鉴权设计：让平台容器自如操作代理池

> 阶段定位：把 v0.1.4 的"静态租户 Key 表"升级为**控制面可签发、可撤销、可轮换的动态 Key 体系**，并明确云平台（platform-api）→ ProxyHarbor → 平台容器三方的鉴权契约。
> 仍然不引入 OIDC / JWT 业务令牌、不引入细粒度 RBAC scope、不做配额限流（这些放 v0.1.6+）。
> 工作量目标：净增 ≤ 1200 行（含测试），不动 selector / gateway forward 主路径。
> 发布形态：单 binary、新增对 MySQL `tenants` / `tenant_keys` 表的依赖（已有的 storage 通道复用），无新中间件。

---

## 0. 设计哲学

| 维度 | 取舍 | 理由 |
| --- | --- | --- |
| **身份分层** | 三种主体：Bootstrap Admin、Tenant API Key、Lease Credential | 管理面、控制面、数据面各管一层，互不串号；任何一层泄漏不会越权到上层 |
| **Key 来源** | DB 表为唯一真相源；env 仅注入 Bootstrap Admin Key | v0.1.4 的 `PROXYHARBOR_TENANT_KEYS` 不可在线增删；v0.1.5 必须支持平台动态创建/销毁租户 |
| **缓存模型** | 启动期全量装载 + 运行期"version watermark" 增量刷新（5s 轮询 / pub-sub 可选） | 兼顾 O(1) 反查与"撤销立即生效"；不强依赖 Redis pub/sub，单 binary 仍可用 |
| **零信任注入** | 平台容器拿到的是已绑定 `tenant_id` 的 Key，本身不能选择身份 | 与 v0.1.4 不变量一致：身份由服务端反查决定 |
| **Key 存储** | 仅存 `sha256(pepper \|\| key)`，明文只在签发时回传一次 | DB 泄漏不等于身份泄漏；与 lease 凭据 hash 策略一致 |
| **可撤销** | 软删除 + `revoked_at` + 缓存版本号；撤销 ≤ 5 秒全集群生效 | 容器销毁后 Key 必须能被立即吊销，避免长尾凭据滥用 |
| **审计闭环** | 每次签发/撤销/认证失败都落 `audit_events`；`key_fp` 贯穿日志 | 事后可凭 `key_fp` + DB 历史还原"哪个容器在哪一刻调用了什么" |
| **失败隔离** | DB 不可用时，已加载缓存继续提供服务（fail-open for reads, fail-closed for writes） | 控制面故障不应连带把数据面的网关流量打挂 |
| **不做的事** | 不做 OIDC、不做 JWT 业务令牌、不做 RBAC scope、不做 per-key 配额 | 控制范围；这些是 v0.1.6 / v0.2.x 的事 |

**核心不变量（继承 v0.1.4 并强化）**：

1. `principal.TenantID` 只能由服务端从 Key（或 lease 凭据）反查得到，**任何客户端声明都不能改变它**。
2. Key 撤销后 ≤ 5 秒，所有实例认证失败；已 forward 的连接由 lease 自身 TTL 兜底。
3. 平台容器**永远不持有跨租户的 Key**；一个容器的 Key 一定且只能映射到一个 tenant。
4. Bootstrap Admin Key 只能通过 env 注入，**不能通过任何 API 创建或修改**，避免提权。
5. 配置即真相对于"系统级开关"成立；对于"租户身份"则迁移到 DB 即真相。

## 1. 三层身份模型

```
┌────────────────────────────────────────────────────────────────────┐
│  L0  Bootstrap Admin Key      env: PROXYHARBOR_ADMIN_KEY           │
│      ─ 唯一，启动期注入，不可在线修改                                │
│      ─ 调用 /admin/* 管理租户、签发/撤销 Tenant Key、看全局指标       │
├────────────────────────────────────────────────────────────────────┤
│  L1  Tenant API Key           DB: tenant_keys                      │
│      ─ 由 Admin（或被授权的 platform-api）签发                       │
│      ─ 绑定到唯一 tenant_id                                         │
│      ─ 调用 /policies, /proxies, /leases 等控制面接口               │
│      ─ 可在线轮换 / 撤销 / 设置过期时间                              │
├────────────────────────────────────────────────────────────────────┤
│  L2  Lease Credential         Redis + DB: leases                   │
│      ─ 由 Tenant Key 调用 /leases 创建（v0.1.3 起已有）             │
│      ─ 短期，TTL ≤ 30min（默认）                                    │
│      ─ 仅用于网关 forward / CONNECT，不能调用控制面                  │
└────────────────────────────────────────────────────────────────────┘
```

**主体能力矩阵**（"是否能调用"）：

| 接口 | L0 Admin | L1 Tenant | L2 Lease |
| --- | :---: | :---: | :---: |
| `POST /admin/tenants` | ✅ | ❌ | ❌ |
| `POST /admin/tenants/{id}/keys` | ✅ | ❌ | ❌ |
| `DELETE /admin/tenants/{id}/keys/{kid}` | ✅ | ❌ | ❌ |
| `GET /admin/metrics` | ✅ | ❌ | ❌ |
| `POST /policies` (own tenant) | ✅* | ✅ | ❌ |
| `GET /proxies` (own tenant) | ✅* | ✅ | ❌ |
| `POST /leases` | ✅* | ✅ | ❌ |
| `GET /forward` / `CONNECT` | ❌ | ❌ | ✅ |

> `✅*` = Admin 调用控制面接口时必须显式带 `X-On-Behalf-Of: <tenant_id>`，否则 400；不允许"匿名管理员"操作租户数据，便于审计。

## 3. 详细实现要点

### 3.1 DB Schema（MySQL）

```sql
-- tenants：租户元数据
CREATE TABLE tenants (
    id           VARCHAR(64)  PRIMARY KEY,                       -- 与 v0.1.4 同字符集 [a-z0-9_-]
    display_name VARCHAR(128) NOT NULL,
    status       VARCHAR(16)  NOT NULL DEFAULT 'active',          -- active|suspended|deleted
    created_by   VARCHAR(64)  NOT NULL,                           -- admin principal id
    created_at   DATETIME(3)  NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at   DATETIME(3)  NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    deleted_at   DATETIME(3)  NULL,
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- tenant_keys：API Key 元数据（不存明文）
CREATE TABLE tenant_keys (
    id          VARCHAR(36)  PRIMARY KEY,                         -- key id (uuid)
    tenant_id   VARCHAR(64)  NOT NULL,
    key_hash    BINARY(32)   NOT NULL,                            -- sha256(pepper || key)
    key_fp      CHAR(8)      NOT NULL,                            -- sha256(key)[:8] for log
    label       VARCHAR(128) NOT NULL DEFAULT '',                 -- e.g. "km-wiki-prod-pod-7f9d"
    purpose     VARCHAR(32)  NOT NULL DEFAULT 'general',          -- general|platform_container|ci
    created_by  VARCHAR(64)  NOT NULL,
    created_at  DATETIME(3)  NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    expires_at  DATETIME(3)  NULL,                                -- NULL = 永久
    revoked_at  DATETIME(3)  NULL,                                -- 软删
    last_seen_at DATETIME(3) NULL,                                -- 异步回写，便于排查"幽灵 Key"
    UNIQUE KEY uk_key_hash (key_hash),
    INDEX idx_tenant (tenant_id, revoked_at),
    INDEX idx_fp (key_fp),
    CONSTRAINT fk_tk_tenant FOREIGN KEY (tenant_id) REFERENCES tenants(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- tenant_keys_version：单行版本号，缓存层轮询此值决定是否增量拉取
CREATE TABLE tenant_keys_version (
    id      TINYINT     PRIMARY KEY,            -- 固定 = 1
    version BIGINT      NOT NULL DEFAULT 0,
    updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
INSERT INTO tenant_keys_version (id, version) VALUES (1, 0);
```

> Pepper 是进程级常量（`PROXYHARBOR_KEY_PEPPER`，启动期注入），用于让 DB 备份单独泄漏时仍无法离线撞 Key。Pepper 轮换需要重签所有 Key，本版本不提供 API（手工迁移）。

### 3.2 缓存层（运行期 O(1) 反查 + ≤ 5 秒撤销时效）

```go
// internal/auth/cache.go
type entry struct {
    tenantID  string
    keyID     string
    expiresAt time.Time
    purpose   string
}

type DynamicStore struct {
    mu        sync.RWMutex
    byHash    map[[32]byte]entry
    version   int64
    store     KeyStore               // 实际 DB 访问
    pepper    []byte
    refresh   time.Duration          // 默认 5s
    fallback  bool                   // DB 不可用时使用本地缓存
}

func (d *DynamicStore) Lookup(presented string) (entry, bool) {
    h := sha256Pepper(d.pepper, presented)
    d.mu.RLock(); defer d.mu.RUnlock()
    e, ok := d.byHash[h]
    if !ok { return entry{}, false }
    if !e.expiresAt.IsZero() && time.Now().After(e.expiresAt) { return entry{}, false }
    return e, true
}

// background goroutine：每 refresh 秒读 version，若变化则增量拉取
func (d *DynamicStore) loop(ctx context.Context) { /* ... */ }
```

**关键策略**：

- **读路径** 0 I/O：仅本地 map 查询，与 v0.1.4 同量级延迟。
- **撤销时效**：写 DB → 递增 `tenant_keys_version` → 各实例下一轮 ≤ refresh 秒读到新版本号 → 拉取 `WHERE updated_at > last_seen_version_ts` 的增量。最坏 5s。
- **降级**：DB 故障时 `loop` 跳过；已加载的 Key 继续生效（`fallback=true` 写入指标 `auth_cache_stale_seconds`），避免认证子系统连带故障。
- **冷启动**：启动时全量 `SELECT ... WHERE revoked_at IS NULL`，失败则 fail-fast。

### 3.3 Authenticator 改造

```go
type AuthMode string
const (
    ModeLegacy       AuthMode = "legacy-single-key"   // v0.1.3
    ModeTenantKeys   AuthMode = "tenant-keys"         // v0.1.4 静态
    ModeDynamicKeys  AuthMode = "dynamic-keys"        // v0.1.5 DB
)

type Authenticator struct {
    mode      AuthMode
    legacy    string             // ModeLegacy
    static    map[string]string  // ModeTenantKeys
    dynamic   *DynamicStore      // ModeDynamicKeys
    adminKey  string             // 三种模式下都可启用，独立判定
}

func (a *Authenticator) Authenticate(r *http.Request) (domain.Principal, error) {
    presented := r.Header.Get(HeaderName)
    if presented == "" { return zero, domain.ErrAuthFailed }

    // L0 Admin 优先：常数时间比较
    if a.adminKey != "" && subtle.ConstantTimeCompare([]byte(presented), []byte(a.adminKey)) == 1 {
        return adminPrincipal(r) // 见 3.4
    }

    switch a.mode {
    case ModeDynamicKeys:
        e, ok := a.dynamic.Lookup(presented)
        if !ok { return zero, domain.ErrAuthFailed }
        if claimed := r.Header.Get(TenantHeaderName); claimed != "" && claimed != e.tenantID {
            return zero, domain.ErrTenantMismatch
        }
        return domain.Principal{
            ID:       "tenant-key:" + Fingerprint(presented),
            Type:     "tenant_key",
            TenantID: e.tenantID,
            KeyID:    e.keyID,         // 新增字段，供审计与 last_seen 异步回写
        }, nil
    case ModeTenantKeys:
        // v0.1.4 路径，原样保留
    case ModeLegacy:
        // v0.1.3 路径，原样保留
    }
}
```

### 3.4 Admin 主体 + On-Behalf-Of

Admin Key 命中后产出的 `Principal`：

```go
domain.Principal{
    ID:       "admin:" + Fingerprint(presented),
    Type:     "admin",
    TenantID: "",                      // 留空，后续按 On-Behalf-Of 决定
}
```

中间件 `requireTenant`：

| 主体 | 路由前缀 | 行为 |
| --- | --- | --- |
| Admin | `/admin/*` | `principal.TenantID` 留空允许 |
| Admin | 控制面（`/policies`、`/proxies`、`/leases` 等） | 必须带 `X-On-Behalf-Of: <tenant>`，且 `tenants` 表中该租户必须 `status=active`；否则 400/404 |
| Tenant | `/admin/*` | 403 |
| Tenant | 控制面 | 与 v0.1.4 一致，`X-On-Behalf-Of` 若与 `principal.TenantID` 不一致返回 403 |

> 不允许 Admin "免声明"操作租户数据：所有审计日志都必须落具体的 `tenant_id` + `acted_by_admin=true`，避免事后无法归因。

### 3.5 Key 签发协议

```
POST /admin/tenants/tnt_km_wiki/keys
Authorization: Admin
Body: { "label": "km-wiki-prod-pod-7f9d", "purpose": "platform_container", "ttl_seconds": 0 }

201 Created
{
  "key_id":    "k_01HZ8...",
  "tenant_id": "tnt_km_wiki",
  "key":       "phk_live_8f2c...e1",     // 明文，仅此一次返回
  "key_fp":    "8f2cab19",
  "expires_at": null,
  "created_at": "2026-04-26T15:33:29Z"
}
```

**明文格式约定**：

```
phk_<env>_<32 hex>
     ^^^^^ 防止误用：phk_live / phk_test 在日志 grep 时可识别
           env 取自 PROXYHARBOR_ENV，影响日志染色而非鉴权
```

**安全要点**：

1. 明文只在响应体出现一次；DB 仅存 `sha256(pepper || key)`。
2. 服务端不提供"重新查看明文"的接口；丢失需重发一把新 Key + 撤销旧 Key。
3. `purpose=platform_container` 的 Key 在审计与 metrics 中单独打标，便于运维识别"哪些是被自动化签发的、可以批量回收"。
4. 撤销接口同步写 DB + 异步广播缓存失效（写 DB 即满足正确性，广播只为缩短窗口）。

### 3.6 与 platform-api 的契约

```
┌──────────────┐                ┌─────────────────┐                 ┌──────────────┐
│ Console User │ ──创建实例──▶ │  platform-api   │ ──签发 Key──▶  │ ProxyHarbor  │
└──────────────┘                │ (Bootstrap Adm) │                 │   /admin/    │
                                └─────────────────┘                 └──────────────┘
                                        │
                                        │ 把明文 Key 写入 K8s Secret
                                        ▼
                                ┌─────────────────┐
                                │  km-wiki Pod    │ ──控制面/网关──▶ ProxyHarbor
                                │  env: PHK_KEY   │
                                └─────────────────┘
```

**两次握手**：

1. **租户开通**（实例创建时一次性）
   - platform-api 用 Bootstrap Admin Key 调 `POST /admin/tenants`（如不存在）。
   - 调 `POST /admin/tenants/{id}/keys` 拿到明文 Key。
   - 把 `{ tenant_id, key }` 写入 `Secret/proxyharbor-credentials-<instance>`，挂载为 env：
     ```
     PROXYHARBOR_ENDPOINT=https://proxypool.svc:8080
     PROXYHARBOR_TENANT=tnt_km_wiki
     PROXYHARBOR_KEY=phk_live_8f2c...e1
     ```
   - platform-api 仅持有 Bootstrap Admin Key，**不缓存 Tenant Key 明文**（写完 Secret 即抛弃）。

2. **实例运行期**
   - 容器启动时读 env，带 `ProxyHarbor-Key: $PROXYHARBOR_KEY` 调 ProxyHarbor 控制面/网关。
   - 容器不需要也不能传 `ProxyHarbor-Tenant`（带了也只能 = `tnt_km_wiki`，否则 403）。

3. **实例销毁**
   - platform-api 在 reconciler 删 Pod 之前调 `DELETE /admin/tenants/{id}/keys/{kid}`。
   - 即便 reconciler 失败，Key 也会随 K8s Secret 一起被 GC，但**主动撤销是契约要求**——避免 Key 泄漏后被复用。

### 3.7 容器侧 SDK 约定（非强制，仅推荐）

容器可以零 SDK 直接 `Header.Set("ProxyHarbor-Key", ...)`。如果未来要做 SDK，约定优先级：

```
Key 读取顺序：
  1. env  $PROXYHARBOR_KEY
  2. file /var/run/proxyharbor/key（K8s projected secret）
  3. metadata sidecar（v0.2.x 预留，本版本不实现）
```

### 3.8 错误语义对照表

| 场景 | HTTP | domain error | 日志 reason |
| --- | --- | --- | --- |
| 缺失 `ProxyHarbor-Key` | 401 | `ErrAuthFailed` | `missing_key` |
| Key 不在缓存且 DB 反查无 | 401 | `ErrAuthFailed` | `unknown_key` |
| Key 已撤销（`revoked_at` 非空） | 401 | `ErrAuthFailed` | `revoked_key` |
| Key 已过期 | 401 | `ErrAuthFailed` | `expired_key` |
| Tenant Key 调 `/admin/*` | 403 | `ErrForbidden` | `not_admin` |
| Admin Key 调控制面但缺 `X-On-Behalf-Of` | 400 | `ErrBadRequest` | `missing_on_behalf_of` |
| Admin Key 的 `X-On-Behalf-Of` 指向不存在/已删租户 | 404 | `ErrTenantNotFound` | `tenant_not_found` |
| Tenant Key 声明跨租户 | 403 | `ErrTenantMismatch` | `tenant_mismatch` |
| DB 不可用 + 缓存 miss（冷数据） | 503 | `ErrAuthBackend` | `auth_backend_unavailable` |
| DB 不可用 + 缓存命中 | 200/2xx | — | `auth_cache_stale=true` |

### 3.9 审计事件 schema

补齐 v0.1.4 标记为 no-op 的 `audit_events`：

```json
{
  "ts":         "2026-04-26T15:33:29.123Z",
  "event":      "tenant_key.issued",
  "actor":      { "type": "admin", "id": "admin:7c..." },
  "tenant_id":  "tnt_km_wiki",
  "key_id":     "k_01HZ8...",
  "key_fp":     "8f2cab19",
  "purpose":    "platform_container",
  "request_id": "req_...",
  "remote_ip":  "10.42.3.7",
  "reason":     null
}
```

事件类型：`tenant_key.issued | .revoked | .auth_failed | .auth_succeeded`。
`auth_succeeded` 默认采样（每 `key_fp` 每 60s 一次），避免 hot path 灌满。

## 4. 配置矩阵

| 环境变量 | 必需 | 说明 |
| --- | :---: | --- |
| `PROXYHARBOR_AUTH_MODE` | ✅ | `legacy` / `tenant-keys` / `dynamic-keys`，默认 `dynamic-keys` |
| `PROXYHARBOR_ADMIN_KEY` | ⚠️ | `dynamic-keys` 模式下**必填**；其它模式可选；≥ 32 字符 |
| `PROXYHARBOR_KEY_PEPPER` | ⚠️ | `dynamic-keys` 模式下**必填**；≥ 32 字节随机；轮换需重签所有 Key |
| `PROXYHARBOR_AUTH_REFRESH_INTERVAL` | ❌ | 缓存增量轮询间隔，默认 5s |
| `PROXYHARBOR_AUTH_KEY` | ❌ | legacy 模式专用 |
| `PROXYHARBOR_TENANT_KEYS` | ❌ | tenant-keys 模式专用 |
| `PROXYHARBOR_ENV` | ❌ | `live` / `test`，仅影响新签发 Key 的 prefix |

**模式判定优先级**（`validate()` 一次性确定）：

| AUTH_MODE | TENANT_KEYS | AUTH_KEY | ADMIN_KEY | 结果 |
| --- | --- | --- | --- | --- |
| dynamic-keys | * | * | ✅ | dynamic（其它字段忽略） |
| dynamic-keys | * | * | ❌ | fail-fast |
| tenant-keys | ✅ | * | * | tenant-keys（v0.1.4 行为） |
| legacy | * | ✅ | * | legacy（v0.1.3 行为） |
| 缺省（未设） | ✅ | * | * | tenant-keys |
| 缺省 | ❌ | ✅ | * | legacy |
| 缺省 | ❌ | ❌ | ✅ | dynamic-keys（最现代默认） |
| 缺省 | ❌ | ❌ | ❌ | fail-fast |

## 6. 兼容性与升级路径

### 6.1 兼容矩阵

| 老配置 | v0.1.5 行为 |
| --- | --- |
| 仅 `PROXYHARBOR_AUTH_KEY` | legacy 模式，零行为变化 |
| 仅 `PROXYHARBOR_TENANT_KEYS` | tenant-keys 模式（v0.1.4 路径），零行为变化 |
| 同时设置 `PROXYHARBOR_TENANT_KEYS` 与 `PROXYHARBOR_ADMIN_KEY` | tenant-keys 模式 + Admin 启用（控制面照旧，新增 `/admin/*`） |
| 仅 `PROXYHARBOR_ADMIN_KEY` + `PROXYHARBOR_KEY_PEPPER` | dynamic-keys 模式（推荐） |

### 6.2 推荐升级流程（K8s）

1. 生成 32 字节随机 pepper、32 字节随机 admin key，写入 K8s Secret。
2. 跑 v0.1.5 迁移脚本创建 `tenants` / `tenant_keys` 表。
3. 部署 `PROXYHARBOR_AUTH_MODE=dynamic-keys` + `ADMIN_KEY` + `PEPPER`，**保留** `PROXYHARBOR_TENANT_KEYS` 作为静态后备（可选）。
4. platform-api 接入 Admin API，把现有租户 + Key 一次性导入：
   - 现存 `PROXYHARBOR_TENANT_KEYS` 中的 `tenant:key` 逐项 `INSERT` 到 `tenant_keys`。
   - 校验后将 env 中的 `PROXYHARBOR_TENANT_KEYS` 清空。
5. 后续新建容器走 platform-api 自动签发流程。

### 6.3 回滚

- 配置回滚到 `tenant-keys` 模式 + 原 env 即可，二进制无需回滚（三种模式同 binary 共存）。
- DB 表保留即可，不影响其它模式运行。

## 7. 不在本版本范围（明确延后）

| 项 | 延后到 | 原因 |
| --- | --- | --- |
| OIDC / SSO 接入 | v0.3.x | 需先有 IdP 选型；当前 Admin 仅一把 env Key 已足 |
| 细粒度 RBAC（per-resource scope） | v0.2.x | 当前控制面操作对租户内是平的 |
| 租户级配额 / 限流 | v0.1.6 | 与 selector 吞吐统计耦合，独立设计 |
| Key 自动轮换调度器 | v0.1.6 | 本版本依赖平台侧主动调签发/撤销 API |
| Pepper 在线轮换 | v0.2.x | 需双 pepper 共存机制 |
| mTLS（容器 ↔ ProxyHarbor） | v0.2.x | 与 service mesh 选型耦合 |
| Key 离线撤销列表（CRL） | 不规划 | 缓存 + 版本号已满足 ≤ 5s 时效 |
