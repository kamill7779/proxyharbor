# ProxyHarbor v0.2.0 设计：全局代理池与统一轻量权限模型

> 版本定位：v0.2.0 是一次 breaking reset。它不继续兼容 v0.1.x 的多套 auth mode、历史 migration 链和租户自维护代理池模型，而是建立一个更简单、更云原生、更容易部署和运维的基线。
> 核心原则：平台管理员维护一个全局代理池；租户只消费代理能力；配置尽量少；数据库用一个 init 脚本初始化；不引入 OIDC/JWT/RBAC/组织层级等重型 IAM。

---

## 1. 为什么要重置

v0.1.x 为了逐步演进，形成了三套历史语义：

- `legacy-single-key`：单 key 既能使用也能管理。
- `tenant-keys`：租户 key 能管理自己名下的 provider/proxy。
- `dynamic-keys`：开始区分 admin key 与 tenant key，但仍要背旧模式兼容。

这让权限边界越来越粘：同一个接口在不同模式下含义不同，配置项也不断增加。v0.2.0 直接重置为一套模型：

- Admin 管理平台资源。
- Tenant 使用平台资源。
- Provider/Proxy 是平台全局库存，不属于租户。
- Policy 当前只保留系统默认策略。

这会破坏旧配置，但换来更清晰的权限、安全边界和部署体验。

---

## 2. 非目标

v0.2.0 不做以下事情：

- 不引入 OIDC/JWT 登录体系。
- 不引入复杂 RBAC、scope、组织、项目、角色矩阵。
- 不实现 tenant-specific policy、quota、rate limit。
- 不实现多个代理池分区或标签路由。
- 不提供旧 migration 链的自动升级框架。
- 不保留 `legacy-single-key` / `tenant-keys` / `dynamic-keys` 三套模式兼容。

这些能力可以在后续版本基于新模型增量扩展。

---

## 3. 统一权限模型

系统只保留两类 principal。

### 3.1 Admin

Admin 来自 `PROXYHARBOR_ADMIN_KEY`。

Admin 可以：

- 管理租户。
- 签发、撤销 tenant key。
- 管理全局 provider。
- 管理全局 proxy。
- 管理全局 policy（当前只允许 `default`）。
- 查看 debug / metrics。
- 使用 `X-On-Behalf-Of: <tenant_id>` 代表某个租户创建 lease，用于测试或平台代操作。

### 3.2 Tenant

Tenant 来自数据库签发的 tenant key。

Tenant 可以：

- 创建 lease。
- 续租 lease。
- 撤销自己的 lease。
- 通过 gateway 使用已获得的 lease。

Tenant 不可以：

- 创建、修改、删除 provider。
- 创建、修改、删除 proxy。
- 创建、修改、删除 policy。
- 管理租户或 tenant key。
- 访问 debug / metrics。
- 看到全局 proxy endpoint 列表。

一句话：租户是使用者，不是代理池维护者。

---

## 4. 全局代理池模型

v0.2.0 将 provider/proxy 从租户资源改为平台全局资源。

### 4.1 Provider

Provider 表示代理来源，例如静态导入、云厂商、外部同步源。

- 由 Admin 创建。
- 全局唯一 ID。
- 不再包含 `tenant_id`。
- 租户默认不可见。

### 4.2 Proxy

Proxy 表示一个可调度的上游代理节点。

- 由 Admin 添加。
- 归属于全局 provider。
- 不再包含 `tenant_id`。
- 调度时所有租户共享同一个 healthy proxy pool。
- lease 仍记录 `tenant_id`，用于审计、续租、撤销和后续公平性扩展。

### 4.3 调度边界

候选 proxy 来自全局池，但请求上下文仍包含 tenant：

- 用于审计 lease 属于哪个 tenant。
- 用于未来扩展配额、公平性或标签过滤。
- 当前版本不做复杂隔离，保持简单。

---

## 5. Policy 简化

当前版本只保留一个系统默认策略。

### 5.1 默认策略

初始化脚本 seed：

- `id = default`
- `enabled = true`
- `ttl_seconds = 1800`（可按实现选择合理默认值）

创建 lease 时：

- 不传 `policy_id`：使用 `default`。
- 传 `default`：允许。
- 传其它 policy：拒绝。

### 5.2 Policy 权限

Policy 写操作只允许 Admin。

Tenant 不能创建、修改或删除 policy。v0.2.0 不开放租户自定义策略，避免把权限模型重新复杂化。

---

## 6. API 权限边界

### 6.1 Admin-only API

以下接口只允许 Admin：

- `POST /admin/tenants`
- `PATCH /admin/tenants/{id}`
- `DELETE /admin/tenants/{id}`
- `POST /admin/tenants/{id}/keys`
- `DELETE /admin/tenants/{id}/keys/{key_id}`
- `GET /v1/providers`
- `POST /v1/providers`
- `GET /v1/providers/{id}`
- `PUT /v1/providers/{id}`
- `DELETE /v1/providers/{id}`
- `GET /v1/proxies`
- `POST /v1/proxies`
- `GET /v1/proxies/{id}`
- `PUT /v1/proxies/{id}`
- `DELETE /v1/proxies/{id}`
- `POST /v1/proxies/{id}:health`
- `GET /v1/policies`
- `POST /v1/policies`
- `GET /v1/policies/{id}`
- `PUT /v1/policies/{id}`
- `DELETE /v1/policies/{id}`
- `/debug/*`

### 6.2 Tenant API

以下接口允许 Tenant：

- `POST /v1/leases`
- `POST /v1/leases/{id}:renew`
- `DELETE /v1/leases/{id}`
- gateway 代理请求。

Tenant 不需要直接读取 provider/proxy 列表。底层代理 endpoint 是平台库存信息，不暴露给租户。

## 7. 配置瘦身

v0.2.0 删除旧模式配置，启动配置只保留必要项。

### 7.1 删除的配置

以下配置不再支持：

- `PROXYHARBOR_AUTH_MODE`
- `PROXYHARBOR_AUTH_KEY`
- `PROXYHARBOR_TENANT_KEYS`

不再做 auth mode 自动推断。

### 7.2 必需配置

生产启动必须提供：

- `PROXYHARBOR_ADMIN_KEY`
- `PROXYHARBOR_KEY_PEPPER`
- `PROXYHARBOR_MYSQL_DSN`

缺任意一项，进程启动失败。

### 7.3 可选配置

保留少量运行时配置：

- `PROXYHARBOR_REDIS_ADDR`
- `PROXYHARBOR_REDIS_PASSWORD`
- `PROXYHARBOR_REDIS_DB`
- `PROXYHARBOR_AUTH_REFRESH_INTERVAL`
- `PROXYHARBOR_AUTH_INVALIDATION`
- `PROXYHARBOR_INSTANCE_ID`
- `PROXYHARBOR_LOG_LEVEL`
- `PROXYHARBOR_LOG_FORMAT`
- `PROXYHARBOR_SHUTDOWN_TIMEOUT`

Redis 仍是可选加速组件，不是 auth 正确性的唯一来源。

### 7.4 Helm values

Helm 也同步瘦身。

推荐结构：

```yaml
auth:
  existingSecret: proxyharbor-credentials
  keys:
    adminKey: admin-key
    pepper: pepper

mysql:
  existingSecret: proxyharbor-credentials
  existingSecretKey: mysql-dsn

redis:
  addr: ""
  existingSecret: ""
  existingSecretKey: redis-password
```

删除：

- `auth.mode`
- `auth.key`
- `auth.tenantKeys`
- 默认 `change-me`

Chart 不应靠弱默认密钥启动。开发环境也应显式创建 Secret。

## 8. 数据库初始化

v0.2.0 使用一个初始脚本作为新安装基线：

```text
migrations/mysql/init.sql
```

废弃旧增量链：

- `001_init.sql`
- `002_proxy_health_columns.sql`
- `003_tenants.sql`
- `004_tenant_keys.sql`

### 8.1 必要表

`init.sql` 一次性创建：

- `tenants`
- `tenant_keys`
- `tenant_keys_version`
- `providers`
- `proxies`
- `policies`
- `leases`
- 必要的 lease/audit/health 辅助表或字段

### 8.2 关键变化

- `providers` 不再有 `tenant_id`。
- `proxies` 不再有 `tenant_id`。
- `leases` 保留 `tenant_id`。
- `policies` 当前为全局表，seed `default`。
- `tenant_keys_version` seed `id=1`。

### 8.3 启动校验

启动时校验：

- 必要表存在。
- `tenant_keys_version.id=1` 存在。
- `policies.default` 存在且 enabled。
- MySQL DSN 可连接。

缺失时启动失败，并给出明确错误。

## 9. 云原生部署体验

v0.2.0 的部署目标是简单、明确、可观测。

### 9.1 Secret-first

敏感配置只从 Secret 注入：

- admin key
- key pepper
- MySQL DSN
- Redis password（如有）

不在 values.yaml 中提供明文默认密钥。

### 9.2 Readiness

`/readyz` 表示实例能否接流量：

- MySQL 可用。
- 必要 schema/seed 已通过启动校验。
- dynamic auth cache 已初始化。
- Redis 不作为 auth 必需条件，除非 selector 配置明确要求。

### 9.3 Liveness

`/healthz` 只表示进程存活，不承载复杂诊断信息。

### 9.4 Debug / Metrics

Admin-only debug/metrics 暴露：

- instance id
- auth cache initialized
- auth cache version
- last refresh time
- stale seconds
- refresh success/failure counter

不得泄露：

- plaintext key
- key hash
- pepper
- admin key
- proxy lease password

## 10. 行为示例

### 10.1 Admin 初始化租户

```bash
curl -H "ProxyHarbor-Key: $ADMIN_KEY" \
  -H "Content-Type: application/json" \
  -d '{"id":"tenant-a","display_name":"Tenant A"}' \
  http://proxyharbor/admin/tenants
```

### 10.2 Admin 添加全局代理

```bash
curl -H "ProxyHarbor-Key: $ADMIN_KEY" \
  -H "Content-Type: application/json" \
  -d '{"id":"proxy-1","provider_id":"static-main","endpoint":"http://proxy1.example.com:8080","healthy":true,"weight":100}' \
  http://proxyharbor/v1/proxies
```

不需要 `X-On-Behalf-Of`，因为 proxy 是全局资源。

### 10.3 Tenant 创建 lease

```bash
curl -H "ProxyHarbor-Key: $TENANT_KEY" \
  -H "Content-Type: application/json" \
  -d '{"subject":{"type":"workload","id":"app-1"},"resource_ref":{"kind":"url","id":"https://example.com"}}' \
  http://proxyharbor/v1/leases
```

系统自动使用 `default` policy，并从全局 healthy proxy pool 中选择代理。

## 11. 测试验收

必须覆盖：

- Tenant key 调 `POST /v1/providers` 返回 `403`。
- Tenant key 调 `POST /v1/proxies` 返回 `403`。
- Tenant key 调 `POST /v1/policies` 返回 `403`。
- Admin key 可创建 provider/proxy。
- Admin key 可管理 default policy。
- Tenant key 可创建 lease。
- Lease 调度来自全局 proxy pool。
- Lease 记录正确 `tenant_id`。
- 不传 `policy_id` 时使用 `default`。
- 传非 `default` policy 时拒绝。
- 缺 admin key 启动失败。
- 缺 pepper 启动失败。
- 缺 MySQL DSN 启动失败。
- `init.sql` 初始化后服务可启动。
- 缺 `tenant_keys_version.id=1` 启动失败。
- 缺 `policies.default` 启动失败。
- `/healthz` 保持简单。
- debug/metrics 不泄露 secret。

## 12. 迁移说明

v0.2.0 是 breaking release。

不承诺自动从 v0.1.x schema 原地迁移。推荐路径：

1. 新建数据库。
2. 执行 `migrations/mysql/init.sql`。
3. 配置 admin key、pepper、MySQL DSN。
4. 启动 ProxyHarbor。
5. 用 Admin API 重建 tenants、tenant keys、providers、proxies。
6. 让租户切换到新 tenant key。

如果必须迁移旧数据，应编写一次性离线脚本，而不是把复杂迁移逻辑塞进服务启动路径。

---

## 13. 成功标准

v0.2.0 完成后应满足：

- 配置项更少，没有 auth mode 分叉。
- Provider/Proxy 是全局平台库存。
- 租户不能写代理池资源。
- Policy 当前只有全局 `default`。
- 单个 `init.sql` 可初始化完整 schema。
- 默认部署不包含弱密钥。
- 代码权限判断清晰，不依赖历史模式分支。
- 云原生部署只需要 Secret + MySQL，Redis 可选增强。
