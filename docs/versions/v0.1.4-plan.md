# ProxyHarbor v0.1.4 实施计划：租户机制闭环（Tenant Key Binding）

> 阶段定位：**把"租户"从客户端自报字段升级为服务端可信身份**。不引入 OIDC/JWT，不做 RBAC，不做配额限流，不引入数据库租户表。
> 工作量目标：净增 ≤ 600 行（含测试），不动 storage / selector / gateway forward 主路径。
> 发布形态：单 binary、零外部依赖新增、配置即数据；Compose / K8s / 裸进程一致。

---

## 0. 设计哲学：云原生 · 高可用 · 轻量级

| 维度 | 设计取舍 | 理由 |
| --- | --- | --- |
| **可信来源** | Key 即身份（key → tenantID 反查） | 无需引入 IdP；密钥轮换通过 ConfigMap/Env 滚动重启完成，符合 12-Factor |
| **配置模型** | 环境变量字符串（`tenant:key,tenant:key`） | 与 K8s Secret / Compose env_file 天然兼容；不新增 CRD、不新增表 |
| **多实例一致性** | 配置静态、启动期解析、运行期只读 | 每个实例本地查表 O(1)，无跨实例同步成本，天生 HA |
| **故障域** | 解析失败立即 fail-fast；运行期无 I/O | 不会因为"认证子系统挂了"导致全网关 503 |
| **审计闭环** | `principal.TenantID` 仅来源于 Key 反查 | 日志中的 `tenant_id` 字段事后可证伪 |
| **兼容性** | 保留 `PROXYHARBOR_AUTH_KEY` 作为 legacy single-key 模式 | 老部署可零改动升级，新部署推荐 tenant keys |
| **不做的事** | 不做 Key 轮换 API、不做租户 CRUD、不做 OIDC、不做 RBAC scope | 防止认证模块膨胀；这些放 v0.2.x 平台化阶段 |

**核心不变量**：

1. 启用 tenant keys 后，`principal.TenantID` 只能由服务端从 Key 表反查得到，**任何请求头/请求体声明的 tenant 都不能改变它**。
2. 不存在"匿名调用"：未匹配任何 Key 一律 401；匹配到 Key 但越权声明其他租户一律 403。
3. 配置即真相（Configuration as Truth）：进程内不缓存动态状态，重启即应用新配置。

## 2. 详细实现要点

### 2.1 配置层：`PROXYHARBOR_TENANT_KEYS`

**格式**（紧凑、可读、易在 K8s Secret 中维护）：

```
PROXYHARBOR_TENANT_KEYS=tenantA:keyA1B2C3...,tenantB:keyB4D5E6...
```

**解析规则**：

1. 以 `,` 分割条目，以第一个 `:` 分割 tenant 与 key（key 中允许含 `:`，tenant 不允许）。
2. tenant 必须通过现有 `auth.ValidTenantID`（`[a-z0-9_-]{1,64}`）。
3. key 长度 ≥ 16，避免误用弱密钥；可通过 `PROXYHARBOR_TENANT_KEY_MIN_LEN` 调整（默认 16）。
4. **唯一性**：同一 key 不能映射到多个 tenant；同一 tenant 不能出现两次（避免歧义）。违反 → 启动失败。
5. 解析后构造两个不可变 map：`keyToTenant map[string]string` 与 `tenants map[string]struct{}`。

**Config 结构新增**：

```go
type Config struct {
    // ... 现有字段
    AuthKey    string            // legacy 单 key（保留）
    TenantKeys map[string]string // key -> tenantID；非空时启用 tenant-key 模式
}
```

**模式判定**（在 `validate()` 中确定，运行时不再变化）：

| 条件 | 模式 | 行为 |
| --- | --- | --- |
| `TenantKeys` 非空 | **tenant-keys** | 严格模式，下文规则 2.2 全部生效 |
| `TenantKeys` 为空 且 `AuthKey` 非空 | **legacy-single-key** | 保持当前 v0.1.3 行为（兼容） |
| 两者都为空 | **fail-fast** | 启动失败 |
| 两者都非空 | **fail-fast** | 拒绝歧义配置，提示选其一 |

### 2.2 认证层：Key → Tenant 反查

改造 [internal/auth/auth.go](../../internal/auth/auth.go)：

```go
type Authenticator struct {
    legacyKey   string            // 兼容模式
    keyToTenant map[string]string // 严格模式
    strict      bool              // 是否启用严格模式
}

func New(legacyKey string, tenantKeys map[string]string) *Authenticator { ... }

func (a *Authenticator) Authenticate(r *http.Request) (domain.Principal, error) {
    presented := r.Header.Get(HeaderName)
    if presented == "" {
        return domain.Principal{}, domain.ErrAuthFailed
    }

    if a.strict {
        tenantID, ok := a.keyToTenant[presented]
        if !ok {
            return domain.Principal{}, domain.ErrAuthFailed // 401
        }
        // 严格模式：若客户端额外声明了 ProxyHarbor-Tenant，必须等于 Key 绑定租户，否则 403
        if claimed := strings.TrimSpace(r.Header.Get(TenantHeaderName)); claimed != "" && claimed != tenantID {
            return domain.Principal{}, domain.ErrTenantMismatch // 新错误：403
        }
        return domain.Principal{
            ID:       "tenant-key:" + fingerprint(presented),
            Type:     "tenant_key",
            TenantID: tenantID,
        }, nil
    }

    // legacy: 与 v0.1.3 保持完全一致
    if presented != a.legacyKey {
        return domain.Principal{}, domain.ErrAuthFailed
    }
    tenantID, ok := ResolveTenantID(r)
    if !ok {
        return domain.Principal{}, domain.ErrAuthFailed
    }
    return domain.Principal{ID: "configured-key", Type: "header_key", TenantID: tenantID}, nil
}
```

**关键点**：

- `fingerprint(key) = hex(sha256(key))[:8]`，**仅用于日志/Principal.ID 标识**，不能用于反查。
- `ErrTenantMismatch` 在 [internal/shared/domain/errors.go](../../internal/shared/domain/errors.go) 新增，gateway/server 层映射为 HTTP 403。
- `ResolveTenantID` 在严格模式下**不再被认证调用**；只保留给"无认证开放接口"（目前没有，预留）。
- Key 比较使用 `subtle.ConstantTimeCompare` 不必要（map 反查已是常数时间路径），但 fingerprint 计算用标准库 `crypto/sha256`。

### 2.3 服务层：基本零改动

[internal/control/service.go](../../internal/control/service.go) 已经全部使用 `principal.TenantID`，无需改动业务逻辑。需要做的只有：

1. 全量审计日志确认 `tenant_id` 字段写入用 `principal.TenantID`，**禁止从请求体 / 请求头二次读取**。
2. 网关 forward 路径（[internal/server/gateway.go](../../internal/server/gateway.go)）的 lease 鉴权链路保持不变：lease 在创建时已经被绑定到正确的 tenant（创建时 principal 来自 Key），forward 时校验 lease 凭据即可。

### 2.4 错误语义对照表

| 场景 | HTTP | domain error | 日志 reason |
| --- | --- | --- | --- |
| 缺失 `ProxyHarbor-Key` | 401 | `ErrAuthFailed` | `missing_key` |
| Key 不在 `TenantKeys` 中 | 401 | `ErrAuthFailed` | `unknown_key` |
| 严格模式下声明了不匹配 tenant | 403 | `ErrTenantMismatch` | `tenant_mismatch` |
| Legacy 模式下 tenant 头格式非法 | 401 | `ErrAuthFailed` | `invalid_tenant_format` |
| 配置解析失败 | 启动失败 | — | stderr + exit 1 |

### 2.5 云原生最佳实践落点

| 实践 | 落地方式 |
| --- | --- |
| **12-Factor Config** | 仅通过 env 注入，不读文件、不读 etcd |
| **K8s Secret 友好** | 单字符串值，可直接放 `data.PROXYHARBOR_TENANT_KEYS`（base64 后） |
| **滚动升级安全** | 新旧实例对同一 key 反查结果一致（配置静态）；若中途修改 Secret，按 K8s Deployment rollout 顺序逐步生效 |
| **零状态依赖** | 认证不依赖 Redis/MySQL，`/readyz` 不必新增检查项 |
| **失败隔离** | 认证模块无 I/O，不可能因下游故障导致 401 风暴 |
| **可观测** | 启动日志一行打印：`auth mode=tenant-keys tenants=[default,tenantA,tenantB] keys=3`（不打印 key 本体） |
| **审计可证伪** | 每条 control 请求日志含 `auth_mode` + `tenant_id` + `key_fp`；事后对账时凭 `key_fp` + 配置历史可还原是谁调用 |

### 2.6 Helm Chart 改动

[charts/proxyharbor/values.yaml](../../charts/proxyharbor/values.yaml) 增加：

```yaml
auth:
  # 二选一：legacySingleKey 或 tenantKeys
  legacySingleKey: ""    # 兼容模式；不推荐
  tenantKeys: {}         # 推荐：{ default: "....", teamA: "....." }
```

[charts/proxyharbor/templates/secret.yaml](../../charts/proxyharbor/templates/secret.yaml) 渲染逻辑：

- `tenantKeys` 非空 → 拼接为 `t1:k1,t2:k2` 写入 `PROXYHARBOR_TENANT_KEYS`，**不**写 `PROXYHARBOR_AUTH_KEY`。
- `legacySingleKey` 非空且 `tenantKeys` 为空 → 仅写 `PROXYHARBOR_AUTH_KEY`。
- 两者都非空 → `helm template` 阶段 `fail`，提示用户。

### 3.3 边界与负面

- 严格模式下 `tenant_id` query 参数被忽略（不再作为身份输入）。
- 严格模式下 `ResolveTenantID` 不在认证路径调用（断言：可以删除调用点而所有 A 类用例仍通过）。
- 启动日志不能包含任何 key 明文（grep 测试或 manual review）。

### 3.4 性能与资源

- 不需要新增 benchmark。Map 反查 O(1)，与 v0.1.3 字符串比较同量级。
- 内存：`map[string]string` 数百到数千条目占用可忽略。

## 5. 兼容性与升级路径

### 5.1 向下兼容矩阵

| 老配置 | v0.1.4 行为 |
| --- | --- |
| 仅 `PROXYHARBOR_AUTH_KEY` 设置 | legacy 模式，零行为变化 |
| `PROXYHARBOR_AUTH_KEY` + 客户端发 `ProxyHarbor-Tenant: X` | legacy 模式仍接受（保持 v0.1.3 行为） |
| 升级到 `PROXYHARBOR_TENANT_KEYS` | 严格模式，客户端必须更新 Key 值，但不需要再传 tenant 头 |

### 5.2 推荐升级流程（K8s）

1. 生成每个租户的新 Key（≥ 16 字符，建议 32 字符随机）。
2. 更新 Helm `auth.tenantKeys`，**不要**同时保留 `legacySingleKey`。
3. 通知调用方切换 `ProxyHarbor-Key`（可选移除 `ProxyHarbor-Tenant` 头）。
4. 滚动升级；旧 Pod 仍持旧配置可用，新 Pod 启用严格模式。
5. 老调用方未切换时会得到 401；通过 `key_fp` 日志快速定位未升级方。

### 5.3 回滚

- 配置回滚即回滚行为；二进制无需回滚。
- 紧急情况下可临时改回 `PROXYHARBOR_AUTH_KEY`，5 分钟内恢复。

## 6. 不在本版本范围（明确延后）

| 项 | 延后到 | 原因 |
| --- | --- | --- |
| 数据库租户表 / 租户 CRUD API | v0.2.x（控制台阶段） | 静态配置已满足当前规模，引入表会带来同步、迁移、备份成本 |
| Key 在线轮换 API | v0.2.x | 滚动重启 + Secret 更新已可实现，新 API 收益不抵复杂度 |
| OIDC / JWT | v0.3.x（平台化） | 需要先有 IdP 选型与会话/刷新令牌的设计，超出本阶段 |
| 租户级 RBAC（role/scope） | v0.3.x | 当前所有 control 操作语义对租户内是平的，不需细分 |
| 租户级配额 / 限流 | v0.2.x | 与 selector / gateway 的吞吐统计耦合，需独立设计 |
| Per-tenant 加密密钥 / 数据加密 | 不规划 | 超出代理池产品形态 |
