# ProxyHarbor v0.4.1 计划：SQLite 单体持久化基线

v0.4.1 从 v0.4.0 tag 签出独立开发，目标是把默认单机体验从“依赖 MySQL 的完整闭环”推进到“一个二进制 + 一个 SQLite 文件”的轻量持久化形态。它不改变 HA 路径，不引入新控制面能力，只补齐单体可靠运行所需的本地持久化。

## 1. 目标

- 提供 `storage=sqlite`，作为单机可靠默认候选。
- 保持 v0.4.0 的 lease、auth、audit、metrics、OpenAPI 行为不回退。
- SQLite 与 MySQL 在核心业务语义上保持一致：租约不会被并发续租复活，过期不可续，明文密码不持久化。
- 保持云原生+轻量级方向：单机无需 MySQL/Redis；HA 仍走 MySQL+Redis，不让 SQLite 承担多实例共享状态。

## 2. 非目标

- 不支持多个 Pod/进程共享同一个 SQLite 文件。
- 不做复杂 migration 框架；本版本只支持空库初始化和 schema version 记录。
- 不改变现有 MySQL schema 与 HA Helm 路径。
- 不新增策略 DSL、quota、rate limit、报表或 Web Console。

## 3. P0：SQLite Store 正确性

### 3.1 Store 接口覆盖

实现 `internal/storage/sqlite.go`，覆盖 v0.4.0 已使用的 Store 能力：

- provider/proxy/policy CRUD。
- lease create/get/update/revoke/list。
- idempotency key。
- usage events。
- audit events。
- dependency check。

### 3.2 Admin Store 覆盖

SQLite 需要支持 admin store 语义：

- tenant CRUD。
- tenant key issue/revoke/list。
- tenant keys version bump。
- admin audit events。

### 3.3 Lease 原子语义

SQLite 的 `UpdateLease` 必须使用条件更新保证：

```sql
WHERE tenant_id = ?
  AND lease_id = ?
  AND generation = ?
  AND revoked = FALSE
  AND expires_at > ?
```

`RowsAffected=0` 后必须重读并区分：

- revoked -> `ErrLeaseRevoked`
- expired -> `ErrLeaseExpired`
- generation mismatch -> `ErrStaleLease`
- missing -> `ErrNotFound`

### 3.4 明文凭据边界

- `CreateLease` 只能在首次响应中返回 plaintext password。
- SQLite 不持久化 `Lease.Password`。
- cache、idempotency replay、admin/list/get 均不得返回明文 password。

## 4. P0：SQLite Schema

新增 `migrations/sqlite/init.sql`，最小表集：

- `schema_version`
- `tenants`
- `tenant_keys`
- `tenant_keys_version`
- `providers`
- `proxies`
- `policies`
- `proxy_leases`
- `proxy_idempotency_keys`
- `proxy_usage_events`
- `proxy_audit_events`

索引要求：

- lease：`tenant_id, lease_id` 唯一。
- idempotency：scope string 唯一。
- audit：`tenant_id, occurred_at, event_id`。
- tenant keys：tenant + key id、key fp。

## 5. P1：配置与启动体验

### 5.1 新增配置

- `PROXYHARBOR_STORAGE=sqlite`
- `PROXYHARBOR_SQLITE_PATH=./proxyharbor.db`

规则：

- `storage=sqlite` 时不需要 MySQL DSN。
- `storage=sqlite` 时不需要 Redis。
- `role=all` 是推荐运行方式。
- `role=controller/gateway` 分离部署不推荐 SQLite。

### 5.2 自动初始化

首次启动时：

- 如果 DB 文件不存在，可以自动创建。
- 如果 schema version 不存在，执行 `migrations/sqlite/init.sql`。
- 如果 schema version 不匹配，启动失败并给出明确错误。

### 5.3 Compose 单体示例

新增或调整 Compose 示例：

- `proxyharbor` 单服务。
- volume 挂载 SQLite 文件目录。
- admin key/pepper 仍使用 env 或 secret。

## 6. P1：文档与云原生边界

README 主路径改为：

- 单机：SQLite。
- 多实例：MySQL + Redis。
- memory：仅 dev/test/demo。

Helm 文档需要明确：

- SQLite 可以用于单副本 + PVC。
- 不支持多副本共享同一个 SQLite PVC。
- 多副本必须切到 MySQL + Redis。

## 7. 并行开发拆分

v0.4.1 可以从 `v0.4.0` tag 签出以下分支并发开发：

### A. SQLite Store

- 负责 `internal/storage/sqlite.go`。
- 负责 SQLite schema。
- 不修改 Helm/README 主文档。

### B. SQLite Admin Store

- 负责 tenant/key/audit admin 能力。
- 可以与 A 共享 schema 文件，但不改业务 service。

### C. Config & Bootstrap

- 负责 config、main wiring、schema init。
- 不修改 Store 业务语义。

### D. Docs & Compose

- 负责 README、Compose、版本文档。
- 等 A/C 的配置名确定后合并。

合并建议顺序：A -> B -> C -> D。

## 8. 验收标准

- `go build ./...` 通过。
- `go test ./...` 通过。
- `storage=sqlite role=all` 可启动。
- SQLite 下可完成：创建 tenant、签发 tenant key、创建 lease、gateway validate、写 audit、重启后数据仍存在。
- SQLite 与 MySQL 在 lease revoke/renew/expired/idempotency 语义上行为一致。
- 不引入 Redis/MySQL 作为单机必需依赖。
