# ProxyHarbor v0.4.3 计划：SQLite 运维与轻量升级闭环

v0.4.3 从 v0.4.0 tag 签出独立开发，目标是补齐 SQLite 单体模式的基础运维能力：schema version、轻量 migration、备份/恢复与数据保留。它不新增控制面业务能力，而是让单体 SQLite 具备长期运行的最低运维闭环。

## 1. 目标

- 让 SQLite 单体可安全升级、备份和恢复。
- 给 audit/usage 提供轻量保留策略，避免本地 DB 无限增长。
- 保持云原生+轻量级原则：不用外部数据库也能长期跑；需要 HA 时仍切 MySQL+Redis。
- 所有能力都服务于可运行代码，不发布“只补测试”的版本。

## 2. 非目标

- 不做跨区域备份。
- 不做在线多节点 SQLite 复制。
- 不做复杂 migration DSL。
- 不做 Web 管理页面。
- 不把 SQLite 推荐给多副本 HA。

## 3. P0：Schema Version 与 Migration

### 3.1 Schema Version 表

SQLite 必须有：

```sql
schema_version(version INTEGER NOT NULL, applied_at TEXT NOT NULL)
```

启动时：

- version 缺失：提示运行 init 或自动初始化。
- version 过低：按 migration 顺序升级。
- version 过高：拒绝启动，提示二进制过旧。

### 3.2 轻量 Migration 目录

建议目录：

```text
migrations/sqlite/
  001_init.sql
  002_*.sql
```

规则：

- migration 必须幂等或由 version gate 控制。
- migration 在事务中执行。
- 失败必须回滚并保留原 DB。

## 4. P0：备份与恢复

### 4.1 Backup

新增：

```bash
proxyharbor backup --output proxyharbor-backup.db
```

要求：

- 使用 SQLite online backup API 或等价安全方式。
- 不直接复制正在写入的 DB 文件。
- 输出不包含明文 admin key、pepper 或 lease password。

### 4.2 Restore

新增：

```bash
proxyharbor restore --input proxyharbor-backup.db
```

要求：

- restore 前检查目标进程未运行或要求显式 `--force`。
- 检查 schema version。
- 恢复后 `doctor` 通过。

## 5. P1：Audit / Usage Retention

新增配置：

- `PROXYHARBOR_AUDIT_RETENTION_DAYS`
- `PROXYHARBOR_USAGE_RETENTION_DAYS`
- `PROXYHARBOR_RETENTION_INTERVAL`

规则：

- 默认不删除，避免惊讶。
- 用户显式配置后定期清理。
- 清理只影响 audit/usage，不影响 tenant/proxy/policy/lease 主数据。
- 清理结果写日志和 metrics。

## 6. P1：SQLite Health 与 Metrics

新增或复用 metrics：

- DB file size。
- migration status。
- retention deleted rows。
- backup/restore success/fail。

注意：

- 不引入高基数 label。
- 不暴露 DB path 中的敏感目录信息。

## 7. P1：文档与 Runbook

新增单体运维 Runbook：

- 初始化。
- 升级。
- 备份。
- 恢复。
- DB 文件权限。
- 容器 volume/PVC。
- 从 SQLite 迁移到 MySQL 的手动路径说明。

## 8. 并行开发拆分

v0.4.3 可以从 `v0.4.0` tag 签出以下分支并发开发：

### A. Migration Engine

- 负责 schema version、migration runner。
- 不实现 backup/restore。

### B. Backup & Restore

- 负责 CLI 和 SQLite backup API。
- 依赖 schema version 读取接口，可先 mock。

### C. Retention Worker

- 负责 audit/usage 清理。
- 不改 lease/core 语义。

### D. Runbook & Metrics

- 负责文档和运维指标。
- 不改存储逻辑。

合并建议顺序：A -> B -> C -> D。

## 9. 验收标准

- `go build ./...` 通过。
- `go test ./...` 通过。
- SQLite 空库可 init，旧 schema 可 migration，新 schema 可启动。
- backup 文件可 restore 到新路径并通过 `doctor`。
- retention 只删除 audit/usage 且可观测。
- 单体路径仍不需要 MySQL/Redis。

## Worker C SQLite Ops Notes

Implemented on `feature/v0.4.3-sqlite-ops`:

- Reusable `database/sql` migration primitives with ordered version gating and `schema_version` support.
- Safe offline `backup`, `restore`, and retention preview command skeletons that work before SQLite Store integration.
- Retention SQL builder and worker functions for `proxy_audit_events` and `proxy_usage_events`.
- Single-node SQLite operations runbook at `docs/runbooks/sqlite-monolith-ops.md`.
