# v0.5.5 - HA Hot Path and Soak Closure

Date: 2026-05-04

## Scope

v0.5.5 stays inside the HA hot-path closure scope:

- reduce `lease_create` write-path waste and connection pressure
- reduce `gateway_validate` cache-hit store rechecks without trusting stale cache blindly
- preserve SSRF / unsafe-target semantics
- preserve v0.5.1-v0.5.4 correctness, runtime, cache, and SDK behavior

This release now meets the formal `500` concurrency / `10m` mixed-soak error-rate gate on the local HA Docker topology. The per-operation latency gates remain above target and are documented as the next performance frontier rather than hidden.

## What Changed

### Control-plane hot path

- `safeTarget` / `safeResource` / `isSafeHost` are now `Service`-owned and use a public-IP literal fast path, short in-process safety decision caching, and in-flight DNS lookup coalescing.
- Domain `ResourceRef` values are never rewritten or persisted as resolved IPs, so dynamic proxy domains remain dynamic.
- `CreateLease` no longer performs service-layer idempotency preflight reads.
- `CreateLease` now uses idempotency-derived lease IDs so MySQL can use the primary-key duplicate path for replay.
- Default policy and selectable proxy reads use short-TTL, singleflight-style caching.
- transient zfair timeout / empty / malformed / stale-selector failures fall back to the local selector instead of failing the request immediately.

### ValidateLease cache/store cooperation

- validate-truth is stored per lease with a fingerprint and a very short freshness window.
- local fast-path validate is allowed only when the corresponding shared Redis truth record still exists and the fingerprint matches.
- renew and revoke invalidate both the distributed lease cache and validate-truth.
- stale-cache correctness is preserved: cache entries without fresh truth still fall through to `store.GetLease(...)`.

### HA test topology

- nginx keeps the HA-pressure tuning:
  - `least_conn`
  - `proxy_next_upstream_tries 1`
  - upstream `max_fails=0`
- MySQL HA-test tuning now uses:
  - `max_connections=1024`
  - `innodb_flush_log_at_trx_commit=2`
  - `skip-log-bin`
  - `innodb_doublewrite=0`
- app MySQL pool defaults in the HA test compose are `64/64` per instance.
- HA tools default to `https://1.1.1.1` instead of `https://example.com` so local DNS pollution does not dominate results.

## Baseline

Baseline evidence carried into v0.5.5:

- PR #20 formal evidence:
  - `gateway_validate`: p95 `265ms`, p99 `269ms`
  - `lease_create`: p95 `808ms`, p99 `828ms`
  - `lease_renew`: p95 `316ms`, p99 `318ms`
  - short soak `10s / 500`: error rate `0.0011`
  - long soak `2m / 500`: error rate `0.9854`, dominated by `502/504`
- branch diagnostics before the final candidate:
  - `500 / 2m` mixed soak with MySQL `max_connections=256`: error rate `2.06%`, dominated by MySQL `Error 1040: Too many connections`
  - raising the HA-test MySQL ceiling to `1024` removed the sustained `500` connection-limit failure mode

## Remaining Bottleneck

The remaining limit is no longer catastrophic availability. It is latency under sustained mixed write pressure:

- `lease_create` and `lease_renew` still carry MySQL write/commit tail latency under 500-way mixed load.
- renew conflicts (`409`) are expected from many workers sharing a finite warmup lease pool.
- residual `504` indicates some requests still exceed nginx's 3s upstream timeout during long mixed soak.
- `gateway_validate` is much healthier than v0.5.4, but p95/p99 still miss the aggressive target when mixed with create/renew pressure.

Recommended next step:

1. add lightweight stage timing around create/renew store calls and selector fallback count
2. reduce renew contention in the runner or make the renew pool larger before judging renew p95
3. evaluate MySQL write-path latency separately from nginx timeout behavior
4. only after direct evidence, consider narrower write-path changes; do not reintroduce a broad create semaphore because A/B showed it worsens p95/p99
