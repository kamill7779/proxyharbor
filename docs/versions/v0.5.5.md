# v0.5.5 — HA Hot Path

## Summary

This release closes the two hot-path bottlenecks identified by the v0.5.4 HA
pressure work without changing any externally observable behavior:

- `safeTarget`/`safeResource`/`isSafeHost` are now `Service`-owned and backed
  by a short-TTL in-process host safety cache. Public IP literals skip DNS via
  `net.ParseIP` + `isInternalIP`. Only resolved-unsafe hostname decisions are
  cached for 15 s; safe hostname targets are re-resolved for each new request
  batch, while concurrent lookups of the same hostname are coalesced so hot
  traffic does not fan out into one DNS request per goroutine. Resolver errors
  and empty results fail closed for the current request and are not cached.
  Unsafe-target rejection (loopback, RFC1918, link-local, multicast, CGNAT,
  IMDS) is unchanged. Lease `ResourceRef` IDs are never rewritten or persisted
  as IPs.
- `ValidateLease` now consults `cache.GetLease` and short-circuits when an
  in-process truth stamp (lease fingerprint of generation, revoked flag,
  `ExpiresAt`, `PasswordHash`, `ResourceRef`, `ProxyID`) verifies the cached
  copy within `validateTruthCacheTTL = 250 ms`. Cache hits without a truth
  stamp, fingerprint mismatches, or expired stamps fall through to the store
  exactly as before. `RenewLease` and `RevokeLease` invalidate both the
  distributed cache entry and the local truth so the next validate sees the
  new state immediately. A successful store read now stamps local truth before
  auth/target validation, so bad-password or bad-target attempts do not clear
  the fast path for subsequent legitimate validates inside the TTL. The truth
  stamp is additionally gated by the cache's lease invalidation version, so a
  processed cross-node lease invalidation forces the next validate to recheck
  store even if the local truth TTL has not expired. `TestValidateLeaseRechecksStoreTruthWhenCacheIsStale`
  still holds because that test seeds the cache without a truth stamp.

The fixes turn the HA topology from "collapse to 502 storm" at 500/2 m into
"0 % error, latency above gate" — the hot path no longer fails, but the
remaining headroom under the 10 minute 500-concurrency soak is bounded by an
infrastructure-tier bottleneck documented at the bottom of this page.

## Remaining Bottleneck

The v0.5.5 hot-path fixes restore stability under burst load (500/1500 RPS
with 0 % error) but the system still collapses under sustained 500-concurrency
load. Successful-sample latency is single-digit ms, which means the surviving
requests are healthy — failure is concentrated on requests blocked at the
admission tier.

Strongest evidence pointer:

- per-instance `PROXYHARBOR_MYSQL_MAX_OPEN=32` (3 instances → 96 total
  connections) versus a sustained ~28 K RPS soak demand,
- nginx `proxy_next_upstream_tries 3` with `proxy_read_timeout 3s` amplifies
  load by up to 3× whenever any single instance slows,
- soak status distribution is `502`-dominated (16.7 M / 17 M) — consistent
  with upstream connection / pool exhaustion plus retry amplification,
- `TestRenewLeaseInvalidatesValidateTruth` and the focused per-op pressure
  results show validate truth caching is functional but per-instance, so
  with three round-robined instances each lease is re-checked against the
  store roughly two thirds of the time.

Recommended next step (not in this PR per "smallest possible fix"):

1. Raise `PROXYHARBOR_MYSQL_MAX_OPEN` per instance (e.g. 32 → 128) and
   reconfirm the soak.
2. If still failing, drop `proxy_next_upstream_tries` to 1 in
   `testdata/ha/nginx.conf` so a slow instance fails its own request
   instead of multiplying the load.
3. If still failing, consider extending the validate truth stamp to be
   shared across instances via Redis with a short TTL.
