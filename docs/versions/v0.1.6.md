# ProxyHarbor v0.1.6 计划：分布式多实例运行时加固

> 阶段定位：在 v0.1.5 已可用的轻量多租户模型上，优先补齐分布式多实例的一致性、启动安全、滚动发布与可观测性。  
> 设计取舍：不引入 OIDC/JWT、复杂 RBAC、组织模型或重型 IAM；继续保持 `tenant + tenant_key + bootstrap admin key` 的轻量模型。  
> 发布原则：每个 milestone 独立可发布、可回滚；默认配置必须保持 v0.1.5 兼容，新增分布式能力优先可选启用。  
> 主目标：多副本部署时，key 创建/撤销、readiness、schema 版本、rolling update 与调试信息都具备明确的安全边界。

---

## 0. 背景与非目标

v0.1.5 已实现动态租户 Key、MySQL-backed tenant/key schema、Admin API、dynamic auth cache、Helm 发布链路。它已经能支撑轻量多租户使用，但多实例部署下仍依赖轮询刷新、readiness 语义较粗、schema 版本未显式校验、缺少多实例一致性测试与操作文档。

v0.1.6 的任务不是把 ProxyHarbor 做成 IAM 平台，而是把现有轻量多租户能力改造成更可靠的云原生多实例运行时。

**非目标：**

- 不新增 OIDC/JWT 登录体系。
- 不新增复杂 RBAC/scope/组织层级。
- 不新增 per-tenant quota/rate limit。
- 不改变 v0.1.5 的默认 legacy-safe Helm 启动行为。
- 不要求 Redis 成为 dynamic auth 的硬依赖；Redis pub/sub 只能作为加速通道，polling 必须保底。

## 1. P0：必须完成

### P0-A：Dynamic auth cache 支持 Redis pub/sub invalidation，polling 保底

**问题**

v0.1.5 的 dynamic auth cache 通过 `tenant_keys_version` 轮询刷新。多实例下 key revoke/create 最终会收敛，但收敛延迟受 `PROXYHARBOR_AUTH_REFRESH_INTERVAL` 限制。对于撤销语义，秒级轮询是可用但不够主动。

**解决方案**

- 增加可选 Redis pub/sub invalidation 通道，例如 channel：`proxyharbor:auth:tenant_keys:invalidate`。
- Admin key create/revoke 成功并 bump version 后 publish invalidation event。
- 每个实例 dynamic auth cache 订阅该 channel；收到 event 后立即检查 version 并执行 full refresh。
- polling 保留为兜底：Redis 不配置、订阅失败、消息丢失时仍按现有 refresh interval 收敛。
- pub/sub 失败不得导致 Admin create/revoke 失败；记录 warn 和指标即可。
- event 内容保持轻量：建议包含 `version`、`reason`、`tenant_id`、`key_id`、`instance_id`，但刷新仍以 DB version 为准，不能信任消息直接改缓存。

**建议落点**

- `internal/auth/cache.go`：为 `DynamicStore` 增加 `Invalidate()` 或 `RefreshNow()` 能力，避免订阅逻辑直接操作内部结构。
- `internal/auth/store.go`：复用 `GetTenantKeysVersion` / `GetTenantKeys`。
- `internal/server/admin_handler.go` 或 admin store 周边：在 create/revoke 后触发 publish。
- `cmd/proxyharbor/main.go`：组装 Redis publisher/subscriber，未配置 Redis 时不启用。
- `internal/config/config.go`：新增最小配置，例如 `PROXYHARBOR_AUTH_INVALIDATION=redis|polling` 或基于 Redis addr 自动启用但可关闭。

**验收标准**

- 无 Redis 时，现有 polling 行为保持不变。
- 有 Redis 时，一个实例 revoke/create key，另一个实例无需等待完整 refresh interval 即可刷新。
- Redis pub/sub 失败时 Admin API 仍成功，cache 最终通过 polling 收敛。
- 有单测覆盖 `RefreshNow`/version gate，有集成或可控 fake pub/sub 测试覆盖多实例 invalidation。

### P0-B：`/readyz` 在 dynamic mode 下检查初始 auth cache 已加载

**问题**

多实例滚动发布时，如果实例在 dynamic auth 初始缓存尚未加载完成前就 ready，流量可能打到无法正确鉴权的实例。

**解决方案**

- dynamic mode 下，readiness 必须包含 auth cache initialized 状态。
- `NewDynamicStore` 已执行初始加载时，可将该状态暴露为 `Initialized()` / `Snapshot()`。
- `/readyz` 保持轻量：只检查进程内状态，不在每次 readiness 请求中同步打 DB。
- 如果初始 load 失败，进程启动应失败或 readiness 保持 false；不要接受流量后再慢慢修。
- legacy/tenant-keys 模式不受影响。

**建议落点**

- `internal/auth/cache.go`：暴露 initialized/cache version/last refresh/error snapshot。
- `internal/server/server.go`：readiness dependency checker 增加 auth dependency。
- `cmd/proxyharbor/main.go`：把 dynamic store 或 auth readiness checker 注入 server。

**验收标准**

- dynamic mode 初始 cache 未加载时 `/readyz` 返回非 200。
- dynamic mode 初始 cache 加载完成后 `/readyz` 返回 200。
- legacy/tenant-keys 模式 readiness 不因 auth cache 检查失败。
- 覆盖 server readiness 单测。

### P0-C：启动时校验 MySQL migration/schema version

**问题**

v0.1.5 依赖 `tenants`、`tenant_keys`、`tenant_keys_version` 等表。多实例滚动升级时，如果新 binary 跑在旧 schema 上，错误会延迟到运行时暴露。

**解决方案**

- 引入最小 schema version 校验，不做重型 migration framework。
- 启动 dynamic mode + MySQL 时，检查必要表与关键列存在。
- 推荐增加轻量表 `schema_migrations` 或使用现有迁移文件约定记录版本；如果不想新增表，至少实现 information_schema 检查。
- 校验失败时启动失败，错误信息指出缺少哪个 migration。
- 回滚策略必须明确：回滚到 v0.1.5 或 legacy/tenant-keys 时，新表可以保留。

**建议落点**

- `internal/storage` 或 `internal/server/admin_store_mysql.go`：新增 schema checker。
- `cmd/proxyharbor/main.go`：dynamic mode 构建 MySQL store 后执行 checker。
- `migrations/mysql`：如采用 `schema_migrations`，新增 migration 文件并文档化。

**验收标准**

- 缺少 `tenants` 表时启动失败。
- 缺少 `tenant_keys_version` 表时启动失败。
- schema 完整时启动成功。
- 校验仅 dynamic mode + MySQL 强制；legacy/tenant-keys 不应被 MySQL schema gate 阻塞。

## 2. P1：应该完成

### P1-A：多实例下 Admin key revoke/create 的一致性测试

**问题**

v0.1.5 有单实例动态鉴权与缓存测试，但缺少明确的多实例 create/revoke 传播测试。

**解决方案**

- 用共享 fake key store + 两个 `DynamicStore` 模拟两个实例。
- create key：实例 A 写入 store 并 bump/publish，实例 B 刷新后应能鉴权。
- revoke key：实例 A revoke 并 bump/publish，实例 B 刷新后必须拒绝。
- 同时覆盖 Redis 不可用时 polling fallback。

**验收标准**

- 测试能稳定复现两个实例 cache 从旧版本到新版本的变化。
- 不依赖真实 Redis/MySQL 的单元测试优先；真实依赖可放集成测试或 CI 可选项。

---

### P1-B：Helm 增加多副本 dynamic mode 示例 values

**问题**

v0.1.5 Helm 默认安全，但缺少多副本 dynamic mode 的清晰示例，用户不容易正确配置 MySQL、Redis、admin key、pepper 与 readiness。

**解决方案**

- 新增示例文件，例如 `charts/proxyharbor/examples/dynamic-ha-values.yaml`。
- 示例覆盖：`replicaCount > 1`、`auth.mode=dynamic-keys`、existingSecret、MySQL DSN secret、Redis addr、readiness/liveness、rollingUpdate 建议。
- 示例必须避免明文 secret。
- README 增加简短引用，不展开过多运维教程。

**验收标准**

- `helm template -f charts/proxyharbor/examples/dynamic-ha-values.yaml` 通过。
- 默认 `values.yaml` 行为保持 legacy-safe。
- 文档说明 Redis invalidation 可选，polling 保底。

---

### P1-C：增加 rolling update / rollback 文档

**问题**

多实例发布时，用户需要知道 migration、配置、滚动升级、回滚之间的顺序，否则容易出现 schema/binary/config 不匹配。

**解决方案**

- README 或 `docs/operations/rolling-update.md` 增加简洁流程。
- 内容覆盖：升级前备份、应用 migrations、部署新版本、观察 readiness、回滚到 legacy/tenant-keys、保留或清理新表。
- 明确发布不变式：旧表可保留；回滚配置优先于 drop table。

**验收标准**

- 文档能指导从 v0.1.5 到 v0.1.6 的升级与回滚。
- 文档与 Helm 示例配置一致。

## 3. P2：可以完成

### P2-A：暴露 auth cache freshness 指标

**问题**

多实例下需要判断每个实例的 auth cache 是否新鲜、是否落后、是否频繁刷新失败。

**解决方案**

- 暴露轻量指标，不强制引入 Prometheus client 也可以先走 JSON/debug endpoint。
- 建议字段：`auth_cache_initialized`、`auth_cache_version`、`auth_cache_last_refresh_unix`、`auth_cache_stale_seconds`、`auth_cache_last_error`、`auth_cache_entries`。
- 如果已有 metrics 风格，遵循现有风格；不要引入大依赖。

**验收标准**

- 能从运行实例看到 cache version 与 last refresh time。
- 刷新失败时能观测到 last error 或 failure counter。

---

### P2-B：增加 instance ID、cache version、last refresh time 到 debug/health 信息

**问题**

排查多实例问题时，需要知道请求打到了哪个实例、该实例 cache 版本是多少。

**解决方案**

- 启动时生成或读取 `PROXYHARBOR_INSTANCE_ID`，默认 hostname + pid/随机短 ID。
- `/healthz` 保持简单存活；`/readyz` 或新增 debug endpoint 返回 instance/cache snapshot。
- 不泄露 key、hash、pepper、admin key。

**验收标准**

- debug/health 输出包含 instance ID、auth mode、cache version、last refresh time。
- 输出不包含任何 secret。

---

### P2-C：压测多实例 key revoke propagation 延迟

**问题**

需要量化 Redis invalidation 与 polling fallback 的传播延迟，避免只凭直觉判断一致性。

**解决方案**

- 增加轻量脚本或测试说明，模拟 N 个实例 cache、create/revoke、统计传播延迟。
- 指标：p50/p95/p99 revoke propagation latency、Redis down fallback latency、refresh failures。
- 不要求纳入默认 CI；可作为 `perf/` 或手工验证脚本。

**验收标准**

- 给出本地可运行命令。
- 产出延迟摘要，能比较 Redis invalidation 与 polling fallback。

## 4. Milestone 划分

每个 milestone 独立可发布、可回滚。

### Milestone A：Auth cache invalidation 与 readiness（P0）

- 实现 `DynamicStore` 可观测 snapshot 与主动刷新接口。
- 实现 Redis pub/sub invalidation，可选启用，polling 保底。
- `/readyz` dynamic mode 检查初始 auth cache loaded。
- 验证：`go test ./internal/auth ./internal/server ./internal/config`，并新增多实例 fake pub/sub 测试。

### Milestone B：Schema gate 与多实例一致性测试（P0/P1）

- 启动时校验 dynamic mode 所需 MySQL schema。
- 覆盖缺表/缺列失败测试。
- 增加 create/revoke 多实例一致性测试。
- 验证：`go test ./...`，MySQL schema checker 单测必须可无真实 MySQL 运行。

### Milestone C：Helm 与运维文档（P1）

- 新增 dynamic HA values 示例。
- 更新 README 或新增 rolling update / rollback 文档。
- 验证：`helm lint charts/proxyharbor` 与 `helm template -f charts/proxyharbor/examples/dynamic-ha-values.yaml`。

### Milestone D：轻量可观测与压测脚本（P2）

- 暴露 auth cache freshness snapshot。
- health/debug 信息包含 instance/cache 状态且不泄露 secret。
- 增加 key revoke propagation 压测脚本或说明。
- 验证：相关单测 + 手工脚本说明。

## 5. 全局验收清单

- `go test ./...` 通过。
- `helm lint charts/proxyharbor` 通过。
- `helm template ph charts/proxyharbor` 通过。
- dynamic mode 无 Redis 时保持 polling 行为。
- dynamic mode 有 Redis 时 create/revoke 能主动触发其他实例刷新。
- `/readyz` 不会在 dynamic auth cache 未初始化时放行流量。
- schema 不完整时 dynamic mode 启动失败，legacy/tenant-keys 不受影响。
- 所有新增 debug/health/metrics 输出不得包含 plaintext key、key hash、pepper、admin key。
