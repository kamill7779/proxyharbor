# v1.0.0 - Release Readiness

Date: 2026-05-05

This document is the v1.0.0 readiness contract. It is not a final release announcement and must not be used as evidence that a `v1.0.0` tag already exists.

## Release Position

v1.0.0 is a hardening release for the product paths that are already implemented and documented. The release can proceed only after the release-candidate runbook passes and the remaining blockers below are closed or explicitly deferred with owner approval.

P0 gates for v1.0.0 are availability, correctness, packaging, API/version consistency, and release readiness. Strict p95/p99 latency under extreme mixed write pressure remains a P1 follow-up unless a later PR provides direct evidence that the latency gates are met.

## Supported Capabilities

- SQLite single-node profile: `storage=sqlite`, `selector=local`, one ProxyHarbor process, local secrets, and standalone operations.
- MySQL + Redis HA profile: multiple ProxyHarbor instances sharing MySQL storage and Redis coordination behind a load balancer.
- Redis zfair selector / cache coordination for HA proxy selection, auth/cache invalidation, and validate-truth cooperation.
- Go SDK baseline for proxy acquisition, lease renew, and disable/delete flows against the HA entrypoint.
- Docker, Docker Compose, Helm, GHCR image, and binary release paths, including the standalone defaults and the documented HA example values.

## Non-Goals

- No SQLite multi-instance shared-state support.
- No built-in MySQL or Redis HA orchestration; operators must provide resilient database and Redis infrastructure.
- No Raft, etcd, Consul, or self-built consensus layer.
- No cloud-vendor benchmark, regional capacity, or external proxy data-plane throughput claims.
- No new product behavior or HA algorithm changes beyond release metadata and automation hardening.

## v0.5.5 HA Evidence Carried Forward

The current v1.0.0 readiness path carries forward the v0.5.5 local HA evidence:

- Final `500` concurrency / `10m` mixed soak: `1,805,539` total requests, `1,800,983` success, `4,556` failures.
- Soak error rate: `0.2523%`, below the `0.5%` availability gate.
- Status distribution: `200=1,200,818`, `201=600,165`, `409=2,121`, `500=52`, `504=2,383`, `502=0`.
- The old sustained `502` cascade is closed in the final 10m soak.
- Targeted runtime, correctness, cache, SDK, build, and HA pressure checks were recorded in `docs/versions/v0.5.5.md`.

## Known Latency Limitation

The `500` concurrency / `10m` soak availability gate is met, but the strict per-operation p95/p99 latency gates are not fully met under heavy mixed write pressure:

| Operation | Final 10m p95 | Final 10m p99 | Status |
|---|---:|---:|---|
| `gateway_validate` | `278ms` | `644ms` | above strict target |
| `lease_create` | `619ms` | `1368ms` | above strict target |
| `lease_renew` | `673ms` | `1292ms` | above strict target |

This is a P1 performance follow-up, not a hidden release success claim. The release notes and RC evidence must continue to distinguish availability from strict latency.

## Release Metadata State

The release metadata is now aligned for the first release candidate:

- `/version` defaults to `version=1.0.0-rc.1` and `stability=release-candidate`.
- Release builds inject `Version` and `Stability` from the tag into Docker images and binary artifacts.
- `charts/proxyharbor/Chart.yaml` reports `version: 1.0.0-rc.1` and `appVersion: 1.0.0-rc.1`.
- `api/openapi.yaml` reports `info.version: 1.0.0-rc.1` and `x-stability: release-candidate`.
- Release automation treats `*-rc.*` tags as GitHub prereleases and does not update the `latest` container tag for them.

## Remaining Final-Release Blockers

These blockers must be resolved before tagging final `v1.0.0`:

- A `v1.0.0-rc.1` tag must pass the release workflow and the manual pressure/soak matrix in `docs/runbooks/release-v1.0.0.md`.
- Final source metadata must be promoted from `1.0.0-rc.1` / `release-candidate` to `1.0.0` / `stable`.
- Release notes must stay RC-first until the release-candidate matrix passes.
- `SECURITY.md` is intentionally deferred to the final phase so supported versions and private disclosure flow match the final v1.0 support policy.

## Readiness Decision

`v1.0.0` can be promoted only after:

- a release-candidate tag is validated with the full matrix in `docs/runbooks/release-v1.0.0.md`;
- version, Helm, OpenAPI, Docker, binary, and release metadata are aligned with the candidate/final tag;
- the release evidence records the met `500` concurrency / `10m` soak availability gate and the remaining latency limitation;
- `SECURITY.md` is added in the final phase, after support and disclosure policy are confirmed.
