# ProxyHarbor v0.1.3 实施计划

> 阶段定位：**单体强能力 + 多实例稳健运行**，不做平台化、不做新算法、不做监控大盘。
> 工作量目标：在不重构核心架构的前提下完成，控制在 ~1500 行净增以内。

---

## 0. 评审结论：原计划 vs 当前阶段定位

| 维度 | 原计划 | 本计划裁剪后定位 |
| --- | --- | --- |
| **分布式** | 多实例隔离 + 状态空窗修复 | 同意，**P0 必做**（fix 1 / fix 2） |
| **云原生** | readyz / 启动 retry | 收敛为 readyz 真实依赖检查 + 单层 retry，不引入 leader election、不做 operator |
| **轻量级** | 不引入 Stream、不做监控系统 | **强化**：日志只用标准库 `log/slog`，禁止引入 prometheus/otel 依赖 |
| **高可用** | Redis 不可用拒启动；lazy recovery | 保持当前策略，**不做内存降级**，否则破坏多实例语义 |
| **有序迭代** | 三档优先级 + 推荐顺序 | 重排为「3 个 milestone」，每个 milestone 独立可发布 |

**核心原则**：v0.1.3 不解决「能不能做更多」，而是解决「现在做的事情能不能被解释、被复现、被排查」。

## 2. 详细实现要点

### 2.1 租户解析（Milestone A，P0）

**问题**：当前 [internal/auth/auth.go](../../internal/auth/auth.go) 中 principal 的 `TenantID` 来源不统一，shared key 模式下默认写死。

**实现**：
1. 在 `Authenticate` 函数后添加 `resolveTenant(r *http.Request) string`：优先取 `ProxyHarbor-Tenant` 头，缺省 `default`。
2. 增加白名单校验：tenant ID 仅允许 `[a-z0-9_-]{1,64}`，非法直接 401。
3. 所有 control-plane handler 改为只信任 `principal.TenantID`，禁止从 body 取 `tenant_id`。
4. 单测：[internal/server/server_test.go](../../internal/server/server_test.go) 增加 tenant A/B 隔离用例。

**风险点**：网关侧 `gatewayCredentials` 已经从 username 中解析 tenant，注意保持兼容，不要让 control 头覆盖网关 username 解析。

### 2.2 zfair ready 空窗自恢复（Milestone B，P0）

**问题**：候选集非空但 Redis ZSET ready 为空，返回 `no_healthy_proxy`。

**实现**：在 [redis_zfair.go](../../internal/control/selector/redis_zfair.go) `Select` 方法中，Lua 脚本返回 `EMPTY_READY` 时：
1. Go 侧用候选列表调用 `EnsureCandidates`（新方法）一次性补齐 ready ZSET。
2. 重新 `EVAL` 选择脚本一次（**仅一次**，避免活锁）。
3. 如仍失败，返回 `ErrReadyEmptyAfterRebuild` 内部错误，外部映射为 `no_healthy_proxy`。

**约束**：rebuild 时使用 `SET NX` + per-tenant lock key（TTL 1 s），避免多实例同时 rebuild 引发抖动。

### 2.3 HTTP 健康分类（Milestone B，P0）

**问题**：[internal/server/gateway.go](../../internal/server/gateway.go) `gatewayHTTP` 中只要 `RoundTrip` 不返回 error 就视为成功。

**实现**：在 `recordProxySuccess` 之前根据 `resp.StatusCode` 分流：

```go
switch {
case resp.StatusCode == 407:
    s.recordProxyFailure(ctx, lease, health.FailureAuth, "upstream_407")
case resp.StatusCode == 429:
    s.recordProxyFailure(ctx, lease, health.FailureProtocol, "upstream_429")
case resp.StatusCode >= 502 && resp.StatusCode <= 504:
    s.recordProxyFailure(ctx, lease, health.FailureConn, "upstream_5xx")
default:
    s.recordProxySuccess(ctx, lease, latency, "http_forward")
}
```

[policy.go](../../internal/control/health/policy.go) 不需要改 enum，复用现有 `FailureAuth/Protocol/Conn`。

### 2.4 CONNECT 认证（Milestone B，P1）

**实现**：在 `gatewayConnect` 拨号上游代理后，如果 `proxyURL.User != nil` 必须主动发送：

```
CONNECT host:port HTTP/1.1
Host: host:port
Proxy-Authorization: Basic <base64(user:pass)>
```

注意 `url.UserPassword` 的密码已被解码，`Password()` 返回明文，直接用。

测试：增加一个 mock 上游代理（仅在测试中），断言收到 `Proxy-Authorization` 头且 base64 内容正确。

### 2.5 结构化日志（Milestone A，P0）

**选型**：标准库 `log/slog`，零额外依赖，符合「轻量级」原则。

**注入方式**：
- `cmd/proxyharbor/main.go` 创建 root logger，按环境变量 `PROXYHARBOR_LOG_FORMAT=json|text`、`PROXYHARBOR_LOG_LEVEL=info|debug` 配置。
- 通过 context 传递：`ctx = logging.WithLogger(ctx, logger.With("tenant_id", tid))`，避免侵入函数签名。
- selector / health / gateway 三个 hot path 各加 ≤ 5 个 log site。

**禁打印字段**（编码到一个 `redact()` 工具函数，并在 PR 中单独 review）：
- proxy endpoint 的 userinfo（用 `proxyURL.Redacted()`）
- lease password / password_hash
- `Authorization` / `Proxy-Authorization` / `ProxyHarbor-Key` 头

### 2.6 selector 错误分类（Milestone A，P1）

新增内部 sentinel：

```go
// internal/control/selector/selector.go
var (
    ErrNoCandidateFromStore       = errors.New("no_candidate_from_store")
    ErrAllCandidatesUnhealthy     = errors.New("all_candidates_unhealthy")
    ErrAllCandidatesDelayed       = errors.New("all_candidates_delayed")
    ErrReadyEmptyAfterRebuild     = errors.New("ready_empty_after_rebuild")
    ErrRedisEvalFailed            = errors.New("redis_eval_failed")
)
```

`Service.CreateLease` 捕获这些 error → 记录日志 → 统一返回 `domain.ErrNoHealthyProxy`，**保持 API 兼容**。

### 2.7 readyz 真实依赖（Milestone C，P0）

[internal/server/server.go](../../internal/server/server.go) 的 `readyz` handler 当前只返回静态 `ready`。改为：

1. `Store.Ping(ctx)`（在 [store.go](../../internal/storage/store.go) 接口增加 `Ping`，memory 直接返回 nil，mysql 调 `db.PingContext`）。
2. selector 增加 `HealthCheck(ctx)`：zfair 实现为 `redis.Ping`，first 实现为 nil。
3. 任一失败返回 503 + JSON `{"status":"degraded","reasons":["mysql_down"]}`。
4. **超时 500 ms**（`context.WithTimeout`），避免 readyz 自己拖死。

### 2.8 启动依赖 retry（Milestone C，P1）

[cmd/proxyharbor/main.go](../../cmd/proxyharbor/main.go) 中包装 MySQL/Redis 初始化：

```
最大 10 次，base 500ms，cap 5s，指数退避 + jitter
总超时不超过 60s，否则 fatal
```

不引入第三方退避库，手写 ~20 行即可。

### 2.9 health recorder drain（Milestone C，P1）

[recorder.go](../../internal/control/health/recorder.go) 已有异步 flush。补充：
1. `Close(ctx)` 方法：停止 ticker → 处理剩余 buffer → 等待最后一次 flush → ctx 超时强退。
2. `cmd/proxyharbor/main.go` SIGTERM 处理时调用 `recorder.Close(ctx with 10s)`。
3. flush 失败 retry：仅对 `context.DeadlineExceeded` / 连接类错误重试 1 次，其余错误直接 drop + log warn。

## 3. 不在本版本范围（明确告知）

- **新调度算法**（P2C / weighted random）：本版只做 zfair 加固。
- **sticky session 真实落地**：保留接口占位。
- **管理后台 / API 文档站**：仍以 README + openapi.yaml 为准。
- **Prometheus / OpenTelemetry**：通过 slog 结构化日志即可，引入 metrics 会显著扩大依赖面。
- **Leader election / 多 controller 协调**：当前所有 controller 实例都是无状态的，依赖 MySQL+Redis 即可水平扩展，无需引入 etcd/raft。
- **Redis Stream 事件总线**：health recorder 直写 MySQL 已够用。

## 4. 验收清单（合并到 main 前必须全绿）

### 功能

- [ ] tenant A 创建的 proxy 不出现在 tenant B 的 `GET /v1/proxies`
- [ ] 多租户压测：每租户至少 100 lease，无串租户记录
- [ ] zfair ready 空窗场景：手动 `redis-cli del` ready ZSET 后下一次请求自动 rebuild 并成功
- [ ] 模拟代理返回 407：3 次后该代理 `circuit_open_until` > now
- [ ] 模拟目标站返回 404：代理 health_score 不变
- [ ] 通过带认证的真实上游代理访问 `https://www.google.com` 成功

### 云原生

- [ ] `kill -9 redis` 后 `/readyz` 在 1 s 内返回 503
- [ ] Redis 恢复后 `/readyz` 在 1 s 内返回 200
- [ ] MySQL 未就绪时启动：app 在 30 s 内重试连上，不直接退出
- [ ] SIGTERM 后 health recorder 完成 drain，日志可见 `health.shutdown.drained events=N`

### 日志

- [ ] 任意一次 `no_healthy_proxy` 在日志中可定位到内部 error_kind
- [ ] 全量日志 grep 无明文密码、无明文 lease password、无 `Proxy-Authorization` header 内容
- [ ] JSON 模式下日志可被 `jq` 直接消费

### 兼容性

- [ ] 所有现有 v0.1.2 API 调用方式不需修改
- [ ] `.env.example` 中新增配置项均有默认值，老用户不设置也能正常启动

## 7. 完成标志

v0.1.3 发布后，ProxyHarbor 在「单体强能力 + 多实例稳健运行」象限达到：

```
可隔离      多租户 catalog / selector / lease 完全独立
可恢复      Redis ready 空窗、依赖临时失联可自动恢复
可观测      slog 结构化日志覆盖 selector / health / gateway 全链路
可解释      任意一次失败可从日志定位内部原因
可压测复现  压测脚本不依赖私密凭据，模板化可重跑
```

下一版本（v0.1.4 及之后）才会进入「多实例协调 / sticky / 多算法」象限，**当前不超前设计**。
