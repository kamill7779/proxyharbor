# ProxyHarbor v0.5.5 HA Hot Path Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Close the remaining v0.5.x HA gap by fixing the smallest proven hot-path bottlenecks in lease target safety and gateway validation, then re-run staircase pressure and long soak until the v0.5.5 performance and stability gates are either met or blocked by a newly evidenced bottleneck.

**Architecture:** Start from the post-PR #20 baseline on `origin/main` commit `66fc523` and keep the existing HA Docker topology, `tools/hapressure`, correctness runners, SDK flow checks, and release evidence format unchanged. Apply only two first-line business-path optimizations before touching infrastructure surfaces: a short-TTL in-process host safety decision cache plus public-IP fast path for target checks, and a bounded in-process `ValidateLease` truth cache that allows fresh cache hits to skip store rechecks without trusting stale distributed cache forever.

**Tech Stack:** Go 1.23, existing HA Docker Compose topology, `tools/hapressure`, MySQL store, Redis hot cache, nginx LB, existing CI and verification tooling.

## Proven and Disproven Diagnoses

### Proven hotspots

- `CreateLease` pays repeated hostname safety cost through:
  - `safeTarget`
  - `extractHost`
  - `isSafeHost`
  - synchronous DNS lookup on every request for hostname targets
- `ValidateLease` hits distributed cache and still unconditionally calls `store.GetLease`, so cache-hit validate traffic is still store-bound.

### Disproven directions

- The issue is not just a tiny proxy pool: `hapressure` seeds `10` proxies.
- The issue is not direct outbound proxy stream latency: `lease create` and `lease renew` do not proxy end-user traffic.
- DNS is part of `lease create`, but not the full explanation for the soak collapse.
- `gateway validate` still has a store hot path after cache hit.

## Design Rules For This Version

### Rule 1: Keep hostname semantics intact

- Do not replace domain targets or proxy endpoints with resolved IPs.
- Do not persist resolved IPs into leases, proxies, or resources.
- Only cache short-lived safety decisions or resolution results in-process.
- Already-literal public IP targets may use a fast path because they do not depend on DNS drift.

### Rule 2: Bounded staleness is acceptable only when explicit and local

- Distributed cache alone must not become source of truth.
- A local freshness window may skip store rechecks only when:
  - the cached lease fingerprint was recently confirmed against store truth on this process
  - the cached lease itself still passes local auth, expiry, revoke, and target checks
- Any stale cache entry without a recent local truth stamp must still recheck store.

### Rule 3: Evidence before broadening scope

- Complete both first-line hotspot fixes before changing LB, DB, logging, or metrics behavior.
- If thresholds still fail, change only the next bottleneck directly pointed to by fresh evidence.

## Hotspot Design A: Target Safety Fast Path

### Current code surface

- `internal/control/service.go`
  - `CreateLease`
  - `ValidateLease`
  - `safeProxyEndpoint`
  - `safeTarget`
  - `extractHost`
  - `isSafeHost`

### Required behavior

- Preserve current unsafe-target semantics.
- Preserve current host extraction semantics.
- Keep empty host, localhost, loopback, private IP, link-local, multicast, unspecified, CGNAT, and metadata IP rejection.
- Fail closed on DNS failure or zero results.
- Do not persist hostname-to-IP rewrites.

### Minimal implementation shape

- Convert target safety helpers from package-level free functions into `Service`-owned behavior where needed.
- Add a small in-process host safety decision cache on `Service`:
  - key: normalized host string
  - value: `safe/unsafe` decision plus expiration time
  - protection: `sync.RWMutex`
  - TTL: short and conservative, in the `5s-30s` range
  - time source: `s.now` so tests can drive expiry deterministically
- Keep a literal-IP fast path:
  - parse host with `net.ParseIP`
  - if it is an IP literal, evaluate via `isInternalIP`
  - do not call DNS
- For hostname targets:
  - consult the short-TTL in-process decision cache
  - on miss or expiry, resolve through a `Service`-owned resolver path
  - cache the final safety decision for the short TTL
- Keep `extractHost` behavior string-based and do not alter `resourceMatchesTarget`.

### Test intent

- Repeated safe hostname requests within TTL should resolve once.
- Public IP targets should not resolve through DNS.
- Cache expiry should force a re-resolve.
- Unsafe hosts must remain rejected.
- Domain resource IDs must remain domains, not rewritten IP strings.

## Hotspot Design B: ValidateLease Store-Recheck Reduction

### Current code surface

- `internal/control/service.go`
  - `ValidateLease`
  - `ValidateGatewayRequest`
- `internal/control/service_test.go`
  - `TestValidateLeaseRechecksStoreTruthWhenCacheIsStale`

### Current correctness trap

- The distributed cache returns `domain.Lease` only.
- `domain.Lease.UpdatedAt` is not cache insertion time, so it is not a usable freshness timestamp by itself.
- A direct cache-hit return would break the existing stale-cache safety contract.

### Required behavior

- Preserve `TestValidateLeaseRechecksStoreTruthWhenCacheIsStale`.
- Preserve auth failure, revoke, expiry, and target mismatch semantics.
- Reduce repeated `store.GetLease` calls for hot validate traffic.
- Do not change the distributed cache interface for this version.
- Do not redesign Redis invalidation for this version.

### Minimal implementation shape

- Add a local in-process validate truth cache on `Service`:
  - key: tenant + lease ID
  - value: recently verified lease fingerprint plus verification time
  - protection: `sync.RWMutex`
  - freshness TTL: short and explicit, in the `100ms-500ms` range
  - time source: `s.now`
- Lease fingerprint must include the fields whose stale drift matters to validate correctness:
  - `generation`
  - `revoked`
  - `expires_at`
  - `password_hash`
  - `resource_ref`
  - optionally `updated_at` if used only as part of the fingerprint, not as freshness time
- Validate flow must become:
  1. reject unsafe target immediately
  2. read distributed cache
  3. if cache miss: fetch store, update cache, stamp local truth, then validate
  4. if cache hit:
     - fail fast locally on password mismatch, revoked, expired, or target mismatch
     - build fingerprint
     - if the same fingerprint has a recent local truth stamp, return cached lease without store
     - otherwise fetch store, refresh distributed cache, refresh local truth stamp, then validate store truth
- Local write paths should clear or overwrite local truth stamps when obviously stale:
  - `RenewLease`
  - `RevokeLease`
  - any helper directly changing lease state inside `Service`

### Why this is acceptable

- A stale distributed cache entry with no recent local truth confirmation still rechecks store.
- The only bounded staleness window is the explicit local truth TTL.
- This keeps the current test contract intact while removing the highest-frequency redundant store reads.

### Test intent

- First validate on a cache hit without a local truth stamp still calls store.
- Second validate on the same fingerprint inside the local truth TTL skips store.
- Once the local truth TTL expires, the next validate rechecks store.
- A stale cached revoked-false lease still rechecks store and returns `ErrLeaseRevoked`.
